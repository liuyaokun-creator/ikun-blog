作为一名刚入行的测试开发工程师，我第一次接触接口测试时，完全不知道该如何下手。记得当时项目上线前，团队临时让我负责一个接口的测试工作，我连 Postman 是什么都不知道。那段时间，我一边翻文档一边试错，最终才慢慢摸索出一些门道。

## 一、Postman 基础使用

刚开始用 Postman 的时候，最大的问题就是不知道怎么构造请求。比如要测试一个获取用户信息的 GET 接口，地址是 `https://api.example.com/users/1`，我就直接在地址栏输入这个 URL，然后点击“Send”。结果返回了 404，但我根本不知道哪里错了。

后来才知道，POST 和 GET 请求的区别，还有参数的传递方式。比如 GET 接口通常通过 URL 参数传参，而 POST 接口则放在 Body 中。举个例子：

```http
GET /users/1 HTTP/1.1
Host: api.example.com
```

或者 POST 请求的示例：

```http
POST /login HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "username": "test",
  "password": "123456"
}
```

## 二、环境变量与脚本

随着测试需求变多，我发现手动填写参数太麻烦了。于是开始学习使用 Postman 的环境变量和预请求脚本。比如，我可以设置一个 `base_url` 变量，方便切换不同环境（开发、测试、生产）。

```javascript
// 预请求脚本中设置环境变量
postman.setEnvironmentVariable("base_url", "https://api.test.com");
```

这样在请求地址里只需要写 `/users/1`，就能自动拼接成完整 URL。

## 三、踩坑经验分享

### 1. 环境变量没生效

有一次我在本地设置了环境变量，但在实际请求中却一直报错。最后才发现是 Postman 没有正确选择当前环境。一定要注意在右上角的环境选择器中选对环境。

### 2. 未处理响应头中的 Token

某次测试登录接口时，返回的 Token 没有被正确保存，导致后续请求一直失败。后来我才意识到需要在“Tests”标签中写代码，将 Token 存入环境变量，供后续使用：

```javascript
// 在 Tests 标签中写
pm.test("Token exists", function () {
    pm.expect(pm.response.json().token).to.not.be.undefined;
    pm.environment.set("auth_token", pm.response.json().token);
});
```

## 四、总结
