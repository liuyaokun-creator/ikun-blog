# 软件测试核心知识点：从新手到老手的踩坑笔记

我第一次接触软件测试，是在大学的时候。当时只是被老师安排做一个简单的“登录功能”测试，结果我连测试用例都写不明白，还因为没注意边界值导致一个bug漏掉了。那会儿我才意识到，测试不是“点点点”，而是有系统方法的。

## 一、测试类型要分清

测试分为很多种，比如**功能测试、性能测试、安全测试、兼容性测试**等。刚入行的时候，我只做了功能测试，后来才知道还有这么多门道。

### 实战代码（Python + unittest）

```python
import unittest

class TestLogin(unittest.TestCase):
    def test_login_success(self):
        # 模拟登录成功的情况
        self.assertEqual(login("user123", "pass123"), True)

    def test_login_fail(self):
        # 模拟登录失败的情况
        self.assertEqual(login("wronguser", "wrongpass"), False)

if __name__ == '__main__':
    unittest.main()
```

> 踩坑提醒：别以为写了测试用例就完事了，测试数据和边界值真的很重要。我之前没考虑空密码，结果上线后被用户试出来了。

## 二、测试用例怎么写？

测试用例不是随便写的。好的测试用例应该包含**前提条件、操作步骤、预期结果**。我之前写过一个“注册功能”的测试用例，只写了“输入正确信息，点击注册”，结果测试时发现页面没跳转，才发现没有考虑网络问题。

### 实战模板

| 测试用例编号 | 测试场景 | 输入 | 预期结果 |
|--------------|----------|------|----------|
| TC001        | 正常注册 | 用户名：test123，密码：123456 | 注册成功，跳转首页 |
| TC002        | 密码错误 | 用户名：test123，密码：12345 | 提示密码错误 |

## 三、自动化测试工具推荐

我一开始用的是**Selenium**，后来发现它太慢了。再后来用了**Postman**做接口测试，效率高了不少。

### 实战代码（Postman 接口测试）

- 请求地址：`/api/login`
- 方法：POST
- Body：
  ```json
  {
    "username": "test",
    "password": "123456"
  }
  ```
- 预期响应：`{"status": "success"}`

> 踩坑提醒：不要依赖Postman的UI，后期要用脚本化测试（如Python + requests）。

## 四、持续集成（CI）的重要性

我以前是手动测试，后来公司引入了Jenkins，自动拉代码、跑测试、部署。这让我真正体会到“自动化”的力量。

### 实战配置（Jenkins + Git + unittest）

在Jenkins中配置了一个Job，每次提交代码都会触发：

```bash
git clone https://github.com/your-repo.git
cd your-repo
python -m unittest discover
```

> 踩坑提醒：测试环境要和生产环境一致，否则测试结果可能不准确。

---

## 总结

软件测试不是简单地“点点点”，而是一门需要逻辑、细心和持续学习的技能。我的经验告诉我，**测试用例要覆盖全面，测试工具要选对，自动化要尽早做**。如果你也是刚入门的测试人，记住：**别怕踩坑，坑里有宝藏。**
